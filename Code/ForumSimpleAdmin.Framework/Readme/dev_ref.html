üìã Overview

Dino Generic Admin is a powerful framework that automatically generates complete CRUD admin interfaces from C# models and controllers. No client-side code required!

‚ú® Zero Client Code

Admin interfaces are auto-generated from your C# models and controllers. No Angular/React/JavaScript needed!

üé® Smart Attributes

Extensive attribute system for field configuration, visibility, containers, tabs, and complex input types.

üîê Role-Based Access

Built-in admin user and role management with DinoAdmin master role and granular permissions.

üì± Platform-Specific

File uploads can be separated by platform (Desktop, Mobile, Tablet, App, Custom platforms).

üîç Smart Search

Advanced filtering with multiple operators and conditions for powerful list views.

üåê Multi-Language

Built-in support for multiple languages with configurable language codes.

Adding a New Entity

Each entity requires THREE components with matching naming:

  1. Entity Class (DB Model) - Pure EF Core entity
  2. Admin Model - Inherits BaseAdminModel
  3. Admin Controller - Inherits DinoAdminBaseEntityController

Example: Item Entity

1. Entity: Item.cs (in BL/Data or BL/Models)

public class Item
{
    public int Id { get; set; }
    public string Name { get; set; }
}

2. Admin Model: AdminItemModel.cs (in Areas/Admin/Models)

public class AdminItemModel : BaseAdminModel
{
    [AdminFieldCommon("ID", readOnly: true)]
    [AdminFieldNumber]
    [VisibilitySettings(showOnCreate: false)]
    [ListSettings]
    public int Id { get; set; }

    [AdminFieldCommon("Name", required: true)]
    [AdminFieldText(maxLength: 100)]
    [ListSettings]
    public string Name { get; set; }
}

3. Controller: AdminItemsController.cs (in Areas/Admin/Controllers)

public class AdminItemsController : DinoAdminBaseEntityController<AdminItemModel, Item, int>
{
    public AdminItemsController() : base("item") { }

    protected override async Task<AdminSegment> CreateAdminSegment()
    {
        return new AdminSegment
        {
            General = new AdminSegmentGeneral
            {
                Name = "Items",
                Priority = 20
            },
            UI = new AdminSegmentUI
            {
                Icon = "shopping-cart",
                IconType = IconType.PrimeIcons,
                ShowInMenu = true
            }
        };
    }

    protected override async Task<ListDef> CreateListDef(string refId = null)
    {
        return new ListDef
        {
            Title = "Items List",
            AllowAdd = true,
            AllowEdit = true,
            AllowDelete = true
        };
    }
}
üí° Naming Convention: Entity Item ‚Üí Admin Model AdminItemModel ‚Üí Controller AdminItemsController (plural for collections)

Base Controller Generic Parameters

DinoAdminBaseEntityController<TModel, TEntity, TId>
  • TModel: Admin Model type (e.g., AdminItemModel)
  • TEntity: Database Entity type (e.g., Item)
  • TId: ID property type (e.g., int, string, Guid)
‚ö†Ô∏è CRITICAL: The ID property type MUST match between Admin Model and Entity. If Entity has public int Id, Admin Model must also have public int Id.

Usage Examples

public class AdminItemModel : BaseAdminModel
{
    // Auto-managed creation timestamp
    [AdminFieldCommon("Created Date", readOnly: true)]
    [AdminFieldDateTime(DateTimePickerType.DateTime)]
    [VisibilitySettings(showOnCreate: false)]
    [SaveDate]
    public DateTime CreateDate { get; set; }

    // Auto-managed update timestamp
    [AdminFieldCommon("Last Updated", readOnly: true)]
    [AdminFieldDateTime(DateTimePickerType.DateTime)]
    [VisibilitySettings(showOnCreate: false)]
    [LastUpdateDate]
    public DateTime UpdateDate { get; set; }

    // Auto-managed user tracking
    [AdminFieldCommon("Updated By", readOnly: true)]
    [AdminFieldNumber]
    [VisibilitySettings(showOnCreate: false)]
    [UpdatedBy]
    public int UpdateBy { get; set; }

    // Manual sort order (enables drag-and-drop)
    [AdminFieldCommon("Order", readOnly: true)]
    [AdminFieldNumber]
    [SortIndex]
    [ListSettings]
    public int SortIndex { get; set; }

    // Archive functionality
    [AdminFieldCommon("Archived", readOnly: true)]
    [AdminFieldCheckbox]
    [VisibilitySettings(showOnCreate: false)]
    [ArchiveIndicator]
    public bool Archived { get; set; }

    // Soft delete functionality
    [AdminFieldCommon("Deleted", readOnly: true)]
    [AdminFieldCheckbox]
    [VisibilitySettings(showOnCreate: false)]
    [DeletionIndicator]
    public bool IsDeleted { get; set; }

    // Parent relationship for filtered lists
    [AdminFieldCommon("Category")]
    [AdminFieldSelect(SelectSourceType.DbType, typeof(Category), "Name", "Id")]
    [ListSettings]
    [ParentReferenceColumn]  // Enables /Items/List?refId=5 filtering
    public int? CategoryId { get; set; }
}

Skip Mapping Attribute

Located in: Submodules/Dino.CoreMvc.Admin/Attributes/SkipMappingAttribute.cs

Use [SkipMapping] to control mapping behavior with granular options:

// Skip mapping completely (both directions)
[SkipMapping]
public string ComputedProperty { get; set; }

// Skip only from database to model
[SkipMapping(skipFromDb: true)]
public string ReadOnlyFromDb { get; set; }

// Skip only from model to database
[SkipMapping(skipToDb: true)]
public string ComputedProperty { get; set; }

// Skip null values when mapping to database (preserve existing values)
[SkipMapping(skipNullValues: true)]
public string? OptionalField { get; set; }

// Skip during import/export operations
[SkipMapping(skipImport: true, skipExport: true)]
public string InternalField { get; set; }

// Full control example
[SkipMapping(
    skipFromDb: false,      // Map from DB to model
    skipToDb: true,         // Don't map from model to DB
    skipNullValues: true,   // Don't overwrite with null
    skipImport: true,       // Skip during import
    skipExport: false      // Include in export
)]
public string? ConditionalField { get; set; }

SkipMapping Parameters

Parameter Default Description
skipFromDb false Skip mapping from database entity to admin model
skipToDb false Skip mapping from admin model to database entity
skipNullValues false Skip mapping null values to existing properties (preserves existing DB values)
skipImport false Skip the property during import operations
skipExport false Skip the property during export operations

Common Use Cases

  • Computed/calculated properties: [SkipMapping(skipToDb: true)] - Read from DB but don't save
  • Read-only properties: [SkipMapping(skipFromDb: false, skipToDb: true)] - Display but don't update
  • Optional fields that shouldn't overwrite: [SkipMapping(skipNullValues: true)] - Only update if value provided
  • Internal/system fields: [SkipMapping(skipImport: true, skipExport: true)] - Hide from import/export

The admin panel includes built-in import and export functionality for Excel, CSV, and PDF formats.

  • CSV files (.csv)

Configuration in ListDef

protected override async Task<ListDef> CreateListDef(string refId = null)
{
    return new ListDef
    {
        // ... other settings ...
        
        // Enable import (Excel/CSV)
        AllowExcelImport = true,
        
        // Enable export formats (Flags enum - can combine multiple)
        AllowedExportFormats = ExportFormat.Excel | ExportFormat.Csv | ExportFormat.Pdf,
        
        // Custom export filename (optional, defaults to controller name)
        ExportFilename = "Items"
    };
}

SkipMapping for Import/Export

Use [SkipMapping] attribute to control which properties are included:

// Skip property from import/export
[SkipMapping(skipImport: true, skipExport: true)]
public string InternalField { get; set; }

// Skip only from import
[SkipMapping(skipImport: true)]
public string AutoGeneratedField { get; set; }

// Skip only from export
[SkipMapping(skipExport: true)]
public string SensitiveData { get; set; }

Import Process

  1. Admin downloads template using DownloadImportTemplate endpoint
  2. Template includes all properties except those marked with [SkipMapping(skipImport: true)]
  3. Admin fills template with data
  4. Admin uploads file via Import endpoint
  5. System validates and imports data in batches (default batch size: 1000)
  6. Returns import result with success/error counts

Export Process

  1. Admin selects export format from list view
  2. System exports all list data (respects current filters)
  3. Properties marked with [SkipMapping(skipExport: true)] are excluded
  4. File is downloaded with timestamp in filename

Permissions

  • Import requires PermissionType.Import permission
  • Export requires PermissionType.Export permission
  • Both are checked automatically by the base controller
üí° When Creating a New Entity: Always ask if import/export support is needed:
  • "Do you need import functionality? (Excel/CSV)"
  • "Do you need export functionality? (Excel/CSV/PDF)"
  • If yes, configure AllowExcelImport and AllowedExportFormats in CreateListDef()
  • Mark internal/system properties with [SkipMapping(skipImport: true, skipExport: true)]

Custom Mapping Methods

Located in: Submodules/Dino.CoreMvc.Admin/Models/Admin/BaseAdminModel.cs

Override these methods on your AdminModel for custom mapping logic:

public class AdminItemModel : BaseAdminModel
{
    // Called BEFORE mapping FROM database entity TO admin model
    // Return false to skip automatic mapping (you handle it manually)
    // Return true to continue with automatic mapping
    public override bool CustomPreMapFromDbModel(dynamic dbModel, dynamic model, ModelMappingContext context)
    {
        // You can cast dbModel to specific type if needed: var entity = (Item)dbModel;
        // Or use ExpandoObject for dynamic properties (see AdminBaseSettings example)
        
        // Example: Skip automatic mapping and handle manually
        if (someCondition)
        {
            model.CustomProperty = dbModel.SomeOtherProperty;
            return false; // Stop automatic mapping
        }
        
        return true; // Continue with automatic mapping
    }

    // Called AFTER mapping FROM database entity TO admin model
    // Use this to modify the model after automatic mapping is complete
    public override void CustomPostMapFromDbModel(dynamic dbModel, dynamic model, ModelMappingContext context)
    {
        // Modify model after automatic mapping
        model.CustomProperty = dbModel.SomeOtherProperty;
        model.ComputedValue = CalculateSomething(model);
    }

    // Called BEFORE mapping FROM admin model TO database entity
    // Return false to skip automatic mapping (you handle it manually)
    // Return true to continue with automatic mapping
    public override bool CustomPreMapToDbModel(dynamic model, dynamic dbModel, ModelMappingContext context)
    {
        // You can cast dbModel to specific type if needed: var entity = (Item)dbModel;
        // Or use ExpandoObject for dynamic properties (see AdminBaseSettings example)
        
        // Example: Skip automatic mapping and handle manually
        if (someCondition)
        {
            dbModel.SomeOtherProperty = model.CustomProperty;
            return false; // Stop automatic mapping
        }
        
        return true; // Continue with automatic mapping
    }

    // Called AFTER mapping FROM admin model TO database entity
    // Use this to modify the entity after automatic mapping is complete
    public override void CustomPostMapToDbModel(dynamic model, dynamic dbModel, ModelMappingContext context)
    {
        // Modify entity after automatic mapping
        dbModel.SomeOtherProperty = model.CustomProperty;
        dbModel.UpdateDate = DateTime.UtcNow;
    }
}
üí° Important Notes:
  • All parameters are dynamic - you can cast them to specific types when needed: var entity = (Item)dbModel;
  • For complex scenarios with dynamic properties, use ExpandoObject (see AdminBaseSettings example)
  • ModelMappingContext provides access to current user ID, mapping options, etc.
  • Returning false from CustomPreMap* methods stops automatic mapping - you must handle everything manually
  • Returning true from CustomPreMap* methods allows automatic mapping to continue
  • CustomPostMap* methods are always called after automatic mapping (if it occurred)
  • See Submodules/Dino.CoreMvc.Admin/Models/Admin/AdminBaseSettings.cs for a complex example using ExpandoObject and JSON serialization

protected override async Task<ListDef> CreateListDef(string refId = null)
{
    return new ListDef
    {
        // ... other settings ...
        
        // Actions to other lists
        Actions = new List<ListAction>
        {
            new ListAction(
                ListActionType.List,                    // Navigate to another list
                "View Categories",
                (new AdminCategoriesController()).Id,
                true                                    // Pass entity ID
            ),
            new ListAction(
                ListActionType.Custom,                 // Custom action
                "Export Data",
                Id,
                false,
                actionName: nameof(ExportAction),
                requireConfirmation: true
            )
        },
        
        // Actions on selected items
        SelfActions = new List<ListAction>
        {
            new ListAction(
                ListActionType.Custom,
                "Bulk Update",
                Id,
                false,
                actionName: nameof(BulkUpdateAction),
                passItemSelection: true,                // Pass selected items
                itemSelectionPropertyName: "ids"
            )
        }
    };
}

üîó Complex Types & Repeaters

Located in: Submodules/Dino.CoreMvc.Admin/Attributes/Complex.cs

Complex types allow you to embed nested objects or collections within your admin models. They can be stored as JSON strings or as separate database entities with foreign key relationships.

Single Complex Type

Use [ComplexType] attribute for single nested objects.

JSON Storage (Simple)

[AdminFieldCommon("Extra Data")]
[ComplexType(
    type: typeof(AdminItemsExtraData),
    storeAsJson: true,                    // Store as JSON string
    containerBehavior: ContainerBehavior.Collapsible,
    defaultCollapsed: false,
    showTitle: true,
    titleText: "Additional Information"
)]
public AdminItemsExtraData ItemsExtraData { get; set; }

// Complex type class
public class AdminItemsExtraData : BaseAdminModel
{
    [AdminFieldCommon("Internal Notes")]
    [AdminFieldTextArea(rows: 3)]
    public string InternalNotes { get; set; }

    [AdminFieldCommon("Priority Level")]
    [AdminFieldSelect(SelectSourceType.Enum, typeof(PriorityLevel))]
    public PriorityLevel Priority { get; set; }
}

EF Relationship (Separate Table)

[AdminFieldCommon("Extra Data")]
[ComplexType(
    type: typeof(AdminItemsExtraData),
    storeAsJson: false,                   // Use EF relationship
    relatedEntity: typeof(ItemsExtraData), // DB entity type
    containerBehavior: ContainerBehavior.None,
    deleteOnRemove: true,                 // Delete when parent is deleted
    cascadeDelete: true                   // Cascade delete child entities
)]
public AdminItemsExtraData ItemsExtraData { get; set; }

Repeater (List of Complex Types)

Use [Repeater] attribute for collections of nested objects.

JSON Storage Array

[AdminFieldCommon("Properties")]
[Repeater(
    type: typeof(ItemProperty),
    storeAsJson: true,                    // Store as JSON array
    minItems: 0,
    maxItems: 10,
    allowReordering: true,                // Enable drag-and-drop
    containerBehavior: ContainerBehavior.Collapsible,
    defaultCollapsed: false,
    showTitle: true,
    titleText: "Item Properties",
    disableRepeaterAddItemButton: false,
    disableRepeaterRemoveItemButton: false,
    repeaterRemoveConfirmation: true
)]
public List<ItemProperty> Properties { get; set; }

[AdminFieldCommon("Sub Items")]
[Repeater(

    storeAsJson: false,                   // Use EF relationship

    minItems: 1,
    maxItems: 5,

    cascadeDelete: true,                  // Cascade delete
    containerBehavior: ContainerBehavior.Tab,
    titleText: "Sub Items"
)]
public List<AdminSubItemModel> SubItems { get; set; }

Advanced Conditional Visibility

For complex conditional logic, override the GetVisibilityRules() method in your controller:

public class AdminProductsController : DinoAdminBaseEntityController<AdminProductModel, Product, int>
{
    protected override void GetVisibilityRules(ConditionalVisibilityRules<AdminProductModel> rules)
    {
        // Simple conditions
        rules.AddGroup()
            .When(x => x.ProductType == ProductType.Physical && x.Weight > 10)
            .Show(x => x.FragileShipping);

        // Complex multi-field conditions
        rules.AddGroup()
            .WhenProperty("ProductType", ProductType.Digital)
            .AndProperty("IsActive", true)
            .ShowProperties("DownloadUrl", "LicenseKey");

        // Conditional based on collection count
        rules.AddGroup()
            .WhenProperty("Tags.Count", 0, ">")  // When Tags collection has items
            .Show(x => x.TagPriority);

        // Multiple conditions with OR logic
        rules.AddGroup()
            .When(x => x.CategoryId == 1 || x.CategoryId == 2)
            .OrWhen(x => x.IsPremium)
            .Show(x => x.PremiumFeatures);
    }
}

Visibility Rule Methods

Method Description Example
When(predicate) Condition using lambda expression .When(x => x.IsActive && x.Type == "Premium")
WhenProperty(property, value) Simple property condition .WhenProperty("Status", "Active")
WhenProperty(property, value, operator) Property condition with comparison .WhenProperty("Quantity", 5, ">")
AndWhen(predicate) Additional AND condition .AndWhen(x => x.IsVerified)
OrWhen(predicate) Additional OR condition .OrWhen(x => x.IsAdmin)
Show(action) Show specified properties .Show(x => x.Discount)
Hide(action) Hide specified properties .Hide(x => x.StandardShipping)
ShowProperties(names) Show properties by name .ShowProperties("Field1", "Field2")
HideProperties(names) Hide properties by name .HideProperties("Field1", "Field2")

Comparison Operators

When using WhenProperty(property, value, operator), you can use these operators:

  • "==" - Equal to (default)
  • "!=" - Not equal to
  • "<" - Less than
  • "<=" - Less than or equal
  • ">" - Greater than
  • ">=" - Greater than or equal
  • "contains" - String contains
  • "starts" - String starts with
  • "ends" - String ends with

Best Practices

üí° Conditional Visibility Tips:
  • Use simple attributes ([ShowIf], [HideIf]) for basic conditions
  • Use controller rules for complex logic involving multiple fields
  • Combine both - use attributes for simple cases and rules for complex scenarios
  • Avoid circular dependencies - don't create conditions that depend on fields that are conditionally hidden
  • Test thoroughly - conditional visibility can create complex user flows
  • Use meaningful names for properties referenced in conditions
‚ö†Ô∏è Important Notes:
  • Conditional visibility works on both create and edit forms
  • Hidden fields are still validated if they have validation rules
  • Complex rules in controllers are evaluated server-side
  • Simple attribute conditions work client-side for better UX
